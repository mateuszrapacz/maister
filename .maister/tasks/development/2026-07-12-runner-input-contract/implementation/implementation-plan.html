Harden the phase-continue Input Contract

Five serialized groups harden the runner, migrate the source contract, rebuild all platform variants, and verify the existing safety guarantees. No source code or generated output is edited while this plan is being created.

29total steps
34focused tests

Highrisk level

TL;DR

Groups 1–3 are serialized because they share the runner and contract harness. Group 4 follows the executable contract so documentation describes final recovery semantics. Group 5 is the only group that writes generated trees, and it does so through make build before make validate.

Key Decisions

  • Hard-cut JSON transport: stdin by default or exactly --input-file PATH; remove legacy flags and --options-json.
  • Use a dependency-free, fixture-defined canonical YAML subset and fail closed before any write.
  • Keep plugins/maister/ authoritative; rebuild Codex, Copilot, Cursor, Kilo, and Kiro outputs only through the existing pipeline.
  • Use one shared executable behavior matrix for the source runner and all five generated runner paths.
  • Use a disabled-by-default, test-only operational fault seam for report and transition failure recovery; it is not a payload field or host API.
  • Preserve actor: advisor|arbiter, confidence: high|medium, idempotency, denylist, atomic writes, and stdout/stderr semantics.

Open Questions / Risks

  • Duplicate keys need a scanner or equivalent recursive parser because JSON.parse loses duplicates.
  • Canonical YAML must reject tabs, duplicate/misplaced anchors, malformed records, duplicate keys, and unsupported constructs without broadening into general YAML.
  • There is no cross-file transaction; retries must regenerate missing reports and apply only a pending transition from durable validated state.
  • Denylist evaluation must occur on retries so a blocked record cannot become a successful reuse.
  • Generated runner locations differ across platforms, so missing paths and behavioral drift must fail the matrix.

Execution order and dependencies

  1. Group 1 — JSON boundary.
  2. Group 2 — canonical state and phase preflight; fixtures and red tests precede mutation code.
  3. Group 3 — durable outcomes and recovery; failure-injection tests are red before recovery changes.
  4. Group 4 — normative/source docs, synchronized to final retry semantics.
  5. Group 5 — build, generated behavior matrix, structural validation, and final gap review.

The runner and shared test harness are serialized across Groups 1–3. Group 4 is disjoint after the executable contract stabilizes. Group 5 is the only generated-output write and uses the existing platform scripts.

Standards compliance

Apply .maister/docs/INDEX.md and the global validation, error-handling, conventions, coding-style, commenting, minimal-implementation, plugin-development, and build-pipeline standards plus testing/test-writing standards. In particular: validate early with allowlists and specific errors; preserve cleanup and exit channels; keep dependencies lean; write behavior tests before implementation; run incrementally; edit only source; and rebuild rather than hand-edit generated variants.

Notes

  • Contract tests use temporary fixture copies and byte snapshots to prove immutability.
  • The fault seam is disabled by default and cannot be activated through JSON payload data.
  • Reports read persisted orchestrator.gate_history; HTML remains a faithful escaped presentation.
  • Preserve existing dirty/untracked user artifacts and do not perform automatic rollback.
  • If implementation reveals a recurring canonical-state safety rule, propose it for standards update after implementation.
