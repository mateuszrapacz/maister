Phase 5 · Specification

Harden the phase-continue Input Contract

Strict JSON ingress, fail-closed canonical state validation, and retry-safe continuation persistence.

12requirements
5source call-sites
6runner paths
Highrisk level
Contents: Summary · Scope · Input contract · State boundary · Mutation flow · Requirements · Verification

TL;DR

Replace the continuation runner's legacy multi-flag interface with one strict JSON decision payload read from stdin by default or from --input-file PATH. Validate the payload and canonical orchestrator-state.yml boundary before mutation, preserve safety/idempotency/atomic persistence, and recover interrupted reports or transitions safely.

Key decisions

  • JSON hard cutover; no legacy flags or compatibility shim.
  • Dependency-free canonical YAML only; unsupported shapes fail closed.
  • advisor|arbiter and high|medium remain the runner enums.
  • Source-only edits are followed by make build regeneration for five variants.

Open questions / risks

  • Cross-file atomicity is unavailable, so retries must be deterministic and state-driven.
  • The strict YAML subset must cover all existing orchestrator state fixtures.
  • Generated runner parity depends on the shared matrix executing every built path.

In scope

  • Source runner input/state/persistence/report/retry behavior.
  • Normative gate reference and five source skills.
  • Executable contract tests and generated-variant rebuild/validation.

Out of scope

  • Decision selection, model APIs, UI, and low-confidence routing.
  • Legacy compatibility, YAML/npm dependencies, or JSON state conversion.
  • Manual edits to generated variants or unrelated refactors.

Canonical input contract

The runner reads one complete JSON object from stdin, or from a complete file selected by the only accepted CLI argument --input-file PATH. Empty input, malformed/non-object JSON, duplicate keys, unknown keys, duplicate transport flags, positional arguments, and legacy flags are errors.

Stdout is reserved for JSON results. Actionable errors use stderr and a non-zero exit code. Meaningful spaces, Unicode, quotes, pipes, and newlines are preserved in decision strings.

Canonical state boundary

After payload validation and before writes, the runner proves that orchestrator.gate_history is exactly once under orchestrator, that records use the canonical schema/statuses, and that root phases is a unique sequence of phase IDs/statuses. Tabs, duplicate keys, duplicate or misplaced anchors, YAML anchors/aliases/tags, malformed records, missing phase membership, and unsupported dependency-free YAML constructs are rejected. Validation failures leave state, reports, and parent directories unchanged.

Decision and mutation flow

  1. Parse transport, duplicate-key-aware JSON, and exact payload schema.
  2. Read/validate state; calculate the existing hash of phase, gate, question, and ordered options.
  3. Reuse a validated terminal record, or atomically persist a new decided/blocked record.
  4. Generate requested reports from persisted gate_history only.
  5. Apply a validated forward transition exactly once when requested.
  6. Return JSON only after requested work completes; later retries recover missing reports/transitions from durable state.

Denylisted gates are always blocked, non-zero, non-transitioning, and remain blocked on retry. Fresh transitions require the current phase to be active, the target to be pending and known, and the transition to be supported.

High-risk boundary: state is the workflow source of truth, and cross-file writes cannot be transactional.

Implementation boundary

Retain and minimally deepen the existing failure, atomic-write, scalar, record, report, denylist, and hashing helpers. Replace the legacy argument parser and extend state validation only enough to prove the specified canonical boundary and recovery states. Do not add a YAML library, compatibility abstraction, or unrelated refactor.

Deliverables are the source runner/tests, normative reference, five source skills, regenerated platform variants, and passing verification commands. Implementation follows specification approval and the subsequent audit/planning gates.
