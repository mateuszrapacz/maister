TL;DR

The canonical role contract lands first; deterministic projection and durable execution-event persistence then run in parallel. Materializer/provenance and the atomic multi-root Kiro lifecycle serialize before exact resolution. Host adapters and workflow configuration form the second safe parallel wave. Gate integration and legacy topology removal follow, with native evidence, release gates, documentation, and full verification last. Coverage is risk-based: at least 120 non-redundant focused scenarios, with no fixed maximum.

Key decisions

  • One checked-in projection contract plus strict parser/manifest code; canonical Markdown remains the sole behavior owner.
  • Target artifacts are generated only inside materializer staging and installed with digest-bound runtime metadata.
  • Receipt/journal v2 is a clean-install multi-root schema; v1 is rejected, not migrated.
  • The shared runtime lives under skills/orchestrator-framework/bin/agent-runtime/.
  • Cursor/Kiro return typed unavailable when exact native selection/identity cannot be enforced or observed.
  • Parity baselines, support claims, release wiring, and docs remain last.

Open risks

  • Codex controls, authentication, models, and JSONL/session observations are version-sensitive.
  • Cursor precedence and Kiro delegated identity observability may remain unavailable.
  • Kiro multi-root ownership must ship atomically across every lifecycle command.
  • Preserve unrelated worktree files; Group 9 deletes exactly .codex/agents/{advisor,arbiter,luna_smoke_agent}.toml by explicit recovery-gate override.
  • Workspace-writing Codex workers remain serialized per checkout.

Frozen implementation contracts

Canonical role & projection

  • Exactly plugins/maister/agents/*.md; no target-owned canonical behavior.
  • role_id = stem = frontmatter.name; workflow ID is exact maister:<role_id>.
  • agent-projection-v1.json owns expected inventory, versions, profiles, transforms, destinations, and separate support inventories.
  • Manifest separates logical ID, adapter ID, and nullable native ID; Codex is codex.exec with null native ID.
  • Canonical JSON, UTF-8/LF, stable modes/order/hashes, deterministic manifest and projected-tree digest.
  • Codex: 28 prompts + role schemas; Cursor: 28 Markdown agents; Kiro: 28 descriptor/prompt pairs.

Multi-root clean install

  • Receipt/journal v2 rejects v1 before mutation; no migration or cleanup branch.
  • managed_roots uses stable IDs and whole_tree | leaf_set; inventory identity is (root_id,path).
  • Kiro adds kiro_native_agents leaf ownership under ~/.kiro/agents.
  • One lock and journal cover all roots/settings; publish only after all verify.
  • Unmanaged collisions fail even on identical bytes; stale leaves require prior exact ownership and no drift.
  • Code 7 preserves diagnostic state as unresolved recovery.

Runtime dispatch & evidence

  • Exact resolver produces one immutable DispatchPlan or typed failure.
  • Per-dispatch locked JSONL: modes 0700/0600, canonical events, monotonic sequence, digest chain, exactly one terminal.
  • Fsync before side effects; pre-spawn failure prevents launch; post-spawn failure cancels when possible and never succeeds.
  • Managed codex exec pins working root, model, effort, sandbox, JSONL, schema, and last-message output.
  • No native Codex custom agents, spawn_agent, V1/V2 switching, app server, natural-language fallback, or root fallback.
  • Gate, idempotency, dispatch, attempt, and canonical role identities remain separate.

Testing

  • Every group starts tests-first; risk and non-redundancy determine count, with no fixed maximum.
  • Group-local runs use only new/directly affected tests; broad release suites run in Group 11.
  • Real 28-role inventory plus small malformed fixtures; exact IDs/mappings/digests/support/closure, not counts alone.
  • Transactional failures compare bytes, modes, links, existence, topology, settings, receipts, journals, and unrelated Kiro agents.
  • passed, failed, and unavailable remain distinct; version presence never proves E5/E6.
