TL;DR

Maister will derive every supported-host agent representation from the same 28 canonical Markdown roles and materialize those projections inside the existing validated transaction boundary. Workflows select exact maister:<role_id> identities through a fail-closed resolver: Codex injects the canonical prompt into a generic subagent, while Cursor and Kiro use deterministic native projections. Kiro native leaves participate in one multi-root, receipt-owned lifecycle that preserves unrelated user agents. Advisor follows the same contracts as every ordinary role; legacy Codex TOML topology is removed without migration compatibility.

Key decisions

  • The 28 canonical Markdown agents are the sole behavior owner.
  • A strict, sorted, versioned IR and manifest bind identity, mappings, and digests.
  • Projection runs inside staging before enumeration, validation, hashing, and provenance.
  • Codex uses injection; Cursor uses generated Markdown; Kiro uses generated descriptor/prompt pairs.
  • Exact dispatch has no aliases, fuzzy matching, built-in, inline, root, or default fallback.
  • Kiro uses narrow receipt-owned leaves inside one multi-root lifecycle.
  • Advisor has no special TOML, permissions, model, adapter, or evidence path.
  • Materialization, E5 discovery, and E6 invocation remain independent proof layers.

Open questions / risks

  • hostCodex spawn/model control, Cursor precedence, and Kiro observed identity vary by version and must fail closed when unprovable.
  • safetyEvery receipt, journal, snapshot, drift, recovery, update, uninstall, verify, and status path must understand the same root registry.
  • evidenceExisting E5 may overstate support because --version currently passes discovery.
  • scopeLegacy TOML and receipt-v1 compatibility are intentionally rejected, not migrated.
  • availabilityE5/E6 may remain unavailable without runtime, authentication, a safe scenario, or observable identity.

In scope

  • Strict 28-role IR, manifest, support inventory, and validation.
  • Deterministic Codex, Cursor, and Kiro staging projections.
  • Projection provenance and receipt binding.
  • Multi-root transaction lifecycle with narrow Kiro leaf ownership.
  • Exact resolver, host adapters, execution records, and workflow bindings.
  • Genuine E5/E6 probes, topology, release gates, CI, and documentation.
  • Removal of both tracked legacy Codex TOML profiles and their special lifecycle during approved implementation.

Out of scope

  • Legacy TOML or receipt-v1 migration, backup, runtime cleanup, or compatibility.
  • New canonical roles or treating support helpers as canonical.
  • A general workflow DSL or universal host file format.
  • Runtime/source-checkout generation or committed behavior copies.
  • Advisor-only permissions, model, sandbox, destination, or adapter behavior.
  • Any fuzzy, natural-language, default, built-in, root, or inline fallback.
  • Ownership of all ~/.kiro or protection from malicious external writers.
  • Native support claims without safe observable scenarios.
  • Unrelated Cursor skill-projection cleanup and all UI/visual work.

Goal

Provide one deterministic, auditable agent-delivery and invocation contract across Codex, Cursor, and Kiro CLI so every canonical role—including Advisor—is transformed, installed, resolved, invoked, and evidenced consistently without weakening existing provenance or byte-exact transaction guarantees.

User stories

Maintainer · one behavior source

Edit one canonical Markdown role and derive every selected host representation deterministically, preventing behavior drift.

Maintainer · honest inventory

Validate canonical and support roles separately so a helper such as explore cannot hide a missing role.

Release maintainer · auditability

Bind projection schema versions and digests into package and receipt provenance to reproduce exact delivered bytes.

Installer · fail before mutation

Reject projection errors, collisions, and unsafe references before any target mutation.

Kiro operator · preserve ownership

Manage only Maister's computed native leaves so unrelated agents remain byte-identical through every lifecycle.

Operator · typed outcomes

Distinguish obsolete installation state, unavailable runtime prerequisites, and observed incorrect behavior.

Workflow operator · exact role

Resolve each maister:<role_id> to exactly one verified representation or fail before dispatch.

Workflow operator · ordinary Advisor

Run Advisor through the same role and execution-record path as every canonical role.

Auditor · separate runtime record

Use append-only records binding requested and observed identity to digests without conflating install and runtime evidence.

Technical approach

canonical Markdown → strict role IR/manifest → staged projector/validator → materializer/provenance → managed-root transaction/receipt
workflow logical ID → exact resolver + active receipt/projection → target adapter → append-only execution record

The role domain does not know mutable installation paths or host invocation. The projector does not inspect installed state. The resolver does not generate files. Adapters cannot bypass manifest/receipt validation. Evidence cannot infer discovery or invocation from materialization.

Canonical identity and manifest

The parser consumes immutable resolved source, preserves the complete instruction body, and extracts only metadata required by active callers. Identity remains narrow: canonical filename/name defines the role, workflows add maister:, and target mappings declare exact external IDs. Canonical and support rows remain separate, and all consumers share normalized identity/path collision rules.

Projection and materialization

Normal assembly establishes trusted inputs in empty staging; projection then emits host roles into the same candidate. Existing enumeration, syntax/reference checks, modes, hashes, inventory, release closure, and receipts cover those exact bytes. Codex keeps prompt resources, Cursor receives generated Markdown, Kiro receives native descriptor/prompt leaves, and support assets remain separately inventoried.

Managed roots and transaction safety

The lifecycle becomes root-registry-driven, not a special Kiro copier. Dedicated roots use whole_tree; shared native roots use leaf_set. One journal snapshots all preconditions before mutation and publishes no receipt until every root verifies. Unmanaged collisions always fail; prior owned leaves require exact drift validation; recovery restores touched leaves and required parent topology without owning unrelated siblings.

Exact runtime dispatch

Workflows request logical IDs without native filenames. Resolution checks grammar, installed state, digests, mappings, collisions, host contract, and model control. Codex receives the complete canonical prompt; Cursor/Kiro receive exact native identities. Required controls must be provable or fail closed. Adapters return common terminal observations and the workflow recorder appends requested and observed identities.

Evidence and support status

Projection and transaction success are prerequisites, not E5/E6. E5 compares observed native IDs to the manifest. E6 invokes two distinguishable ordinary roles plus Advisor and binds results to active digests. Missing safe prerequisites are unavailable; contradictory observations are failed. Provisional packaging never becomes a native support claim.

Clean topology cutover

Legacy Advisor/Arbiter TOML files, reconciliation, init text, special tests, parity rows, and documentation are removed from source during approved implementation. The installer does not scan, back up, migrate, or clean historical runtime files. Persisted receipt-v1 state is rejected before mutation with an actionable error.

Testing and standards

Testing approach

  • Use eight behavioral boundaries: IR/manifest; projection; materializer/provenance; managed-root lifecycle; host/support projections; resolver/adapters/records; E5/E6; topology/release/docs.
  • Add 2–8 focused tests per implementation step group; run only new/directly affected tests per group, then broader suites at integration/final boundaries.
  • Use the real 28-role inventory plus malformed fixtures; assert exact IDs, mappings, digests, and support separation.
  • Compare independent projections for paths, bytes, modes, order, and hashes; prove typed zero-output negative cases.
  • Snapshot every affected root/topology before transaction failures and compare exact restored state plus unrelated Kiro agents.
  • Prove exact grammar, digest/collision/model preflight, no fallback, Advisor equality, and append-only records.
  • Distinguish all evidence outcomes/freshness; version availability cannot satisfy E5/E6.
  • Build each target twice, compare packages, inspect isolation/closure, and run clean extracted lifecycle across all roots.

Standards compliance

  • build-pipeline.md: canonical ownership, explicit targets, reproducible three-host validation.
  • coding-style.md: descriptive names, focused functions, shared identity helpers, no dead compatibility paths.
  • commenting.md: comments only for non-obvious transaction/evidence/host constraints.
  • conventions.md: minimal dependencies, proven support claims, clean release inputs, accurate concurrency boundary.
  • error-handling.md: early typed errors, preserved code-7 diagnostics, no unpinned prerequisite installation.
  • minimal-implementation.md: immediate callers only; no migration, future-host, or generic workflow scaffolding.
  • validation.md: allowlists, containment, collision/provenance/freshness checks, extracted artifact verification.
  • test-writing.md: behavior-focused host seams, byte-exact multi-root recovery, honest unavailable evidence.

Success criteria

  1. Exactly 28 valid roles satisfy stem/name/ID equality; malformed, unsafe, duplicate, or colliding inputs fail before projection.
  2. Advisor is structurally and operationally ordinary; negative topology tests reject prohibited exceptions.
  3. Identical projections have identical paths, bytes, formatting, modes, order, hashes, manifest digest, and tree digest.
  4. Codex has all prompts and no TOML; Cursor has exact 28/28 Markdown; Kiro has exact 28/28 closed pairs; support is separate.
  5. Projection/reference/transform/path/collision failures are typed, mutation-free, and never write source.
  6. Receipts and provenance bind canonical-set, manifest, schema/projector, and projected-tree digests; mismatch fails verification.
  7. Every lifecycle command uses one managed-root registry and journal, including the Kiro native root.
  8. Injected failures restore every managed attribute and leave unrelated Kiro agents byte-identical.
  9. Only receipt-owned unchanged stale leaves are removed; drift and unmanaged collisions make zero changes.
  10. Exact logical requests create validated dispatch plus append-only attempts/terminal outcome; all fallback paths are absent.
  11. Codex injects complete prompts; Cursor/Kiro use exact IDs; unsupported required controls fail closed; Advisor shares the path.
  12. E5 observes expected native inventory; E6 distinguishes two ordinary roles plus Advisor and verifies identity/digests.
  13. Missing prerequisites are unavailable; wrong observations fail; binding changes invalidate old records.
  14. No active legacy TOML lifecycle or compatibility remains; obsolete state fails clearly before mutation.
  15. Deterministic isolated packages for all targets contain canonical/projector inputs, exclude forbidden copies, and pass extracted lifecycle.
