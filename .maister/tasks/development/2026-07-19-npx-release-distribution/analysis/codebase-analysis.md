## TL;DR

The repository already has the two hard pieces this task should preserve: a deterministic, target-aware GitHub release package and a journaled installer that owns all host mutation. It has no npm package manifest, launcher, npm publication path, or npx-specific verification boundary. The safest shape is a thin `@mateuszrapacz/maister` JavaScript ESM distribution adapter that resolves one exact release asset for `codex`, `cursor`, or `kiro-cli`, verifies its transport, digest, source, topology, and E3 bindings fail-closed, extracts only into an invocation-private temporary root, and delegates to `plugins/maister/bin/maister-install.mjs`.

This is complex, high-risk work. The primary language is JavaScript ESM, with Make, YAML, shell, GitHub Actions, npm metadata, and cross-platform archive handling as integration surfaces. The launcher must never write host targets, settings, receipts, journals, backups, locks, or recovery state. Before delegation, release/download/archive failures must leave the operator's home and installer state byte-, mode-, link-, existence-, and topology-identical. After delegation begins, the existing installer owns the result and the launcher forwards it without retrying or translating lifecycle semantics.

`latest` should mean the concrete version selected by npm for `@latest`, then require the matching GitHub release tag and archive; it must not independently select a moving GitHub “latest” release that can differ from the npm package. Exact package selectors remain the reproducible baseline.

## Key Decisions

- Preserve `plugins/maister/bin/maister-install.mjs` as the sole host mutation authority. The launcher may resolve, download, inspect, extract, and clean up, but must not import transaction internals to reproduce lifecycle behavior or mutate target paths itself.
- Treat GitHub Release archives as the payload source of truth. The current release build already emits target-specific `maister-<target>.tar.gz` assets, embeds source provenance and E3, and runs an extracted lifecycle smoke.
- Use a closed target-to-asset mapping: `codex` → `maister-codex.tar.gz`, `cursor` → `maister-cursor.tar.gz`, and `kiro-cli` → `maister-kiro-cli.tar.gz`. Do not accept caller-supplied URLs, arbitrary asset names, another target's archive, a branch checkout, or a nearby version as fallback.
- Make verification layered and fail-closed. Compare locally computed bytes with every available trusted-channel observation—GitHub asset digest, `SHA256SUMS`, SBOM/provenance entries, archive source manifest, source content hash, release version/tag, and E3—while documenting that the current sidecars are unsigned integrity records, not publisher authentication.
- Inspect the archive before extraction and revalidate the extracted tree afterward. Existing `tar` extraction in `release-package.test.mjs:74-76` and `:270-271` is suitable for trusted build-artifact tests, not for untrusted network bytes. The new boundary needs archive-specific path, entry-type, link, size, permission, duplicate, and topology policy.
- Keep state-only commands network-free. This is a required behavior, but the current target overlays do not install `maister-install.mjs`; the npm package therefore needs an explicit control-plane strategy for `status`, `verify`, `uninstall`, `rollback`, and `recover` before implementation starts.
- Coordinate npm and GitHub publication as one release unit. The release must verify that npm package version, Git tag, target archives, source version/commit, and public metadata agree; partial publication and retry behavior need an explicit policy.

## Open Questions / Risks

- **Offline state-only delegation is not yet structurally solved.** `maister-install.mjs` can parse state-only commands without `--source`, but the installed target trees are projections of overlay assets, not copies of the installer control plane. Should the npm package carry the installer and its overlays under the same `plugins/maister/bin/maister-install.mjs` path, or should a verified existing control-plane path be discovered? Shipping a control plane creates a version-alignment and npm-package-scope decision; downloading an archive for state-only commands violates the offline requirement.
- **Archive extraction implementation is open.** Node 22 is used by release CI, but the supported engine floor is not declared. A library or subprocess must provide equivalent controls on macOS, Linux, and Windows for gzip/ustar input, links, modes, special files, duplicate/case-colliding names, limits, and cleanup. A direct `tar -xzf` path is not an acceptable security policy.
- **Remote transport has several independent failure modes.** Decide API versus direct release URLs, HTTPS/host allowlists, redirect limits, response status/content-type checks, timeout, content-length and streamed-byte caps, rate-limit handling, and whether an injected fetch adapter is required for deterministic tests. No failure may silently fall back to source checkout or another release.
- **Unsigned verification is not authentication.** `SHA256SUMS`, SBOM, provenance, and a GitHub asset digest can detect inconsistent observations when fetched through a trusted channel, but an attacker who controls the release channel may replace unsigned sidecars with the archive. Publisher signing, npm trusted publishing/provenance, and the trust model need a separate decision.
- **Publication skew is possible.** GitHub publication can succeed while npm publication fails, and the reverse can happen. Define ordering, idempotent retry rules, what `@latest` means during propagation, and whether an unpaired npm version is withheld, deprecated, or repaired.
- **Output and signal composition need a contract.** `--json` must keep stdout machine-readable; resolution diagnostics belong on stderr or in one documented envelope. `SIGINT`/`SIGTERM` must reach the child, wait for its terminal result, avoid false success, and still clean the exact temporary root.
- **Archive/source/E3 binding must remain single-source.** The source manifest is inside `plugins/maister`, while E3 discovery accepts several locations. The launcher must read and validate the embedded records without manufacturing or rewriting them, then let the installer repeat the binding checks.
- **Ownership references need product/legal separation.** `README.md:40` still shows `github:SkillPanel/maister`, and package-facing metadata retains Skillpanel identity in `plugins/maister/overlays/codex/assets/plugin.json:6-18` and `plugins/maister/overlays/cursor/assets/plugin.json:7-8`. Update distribution-facing URLs, plugin metadata, and npm/release instructions deliberately. `LICENSE:3` is a legal attribution and must not be changed as an incidental publisher migration.

## Current codebase and release path

### Build interface and package producer

`Makefile:27-82` is the build interface. It delegates package, E3, overlay, parity, topology, installation, and validation work to Node entry points; `package` is `release-interface.mjs package` at `Makefile:78-79`. Release callers provide `TARGET`, `SOURCE_COMMIT`, `SOURCE_VERSION`, `SOURCE_DATE_EPOCH`, and E3 inputs through the Make environment rather than through shell-interpolated inline JavaScript. A launcher implementation should not create a second build or packaging path.

`plugins/maister/bin/release-interface.mjs:88-159` builds the package source closure. It copies `common`, `lib`, `bin`, `skills`, `agents`, `commands`, the projection contract, and the selected overlay; it retains only `overlay.yml` and `inventory.yml` for other targets and rejects foreign target assets, legacy behavior trees, and incomplete runtime closure. The required closure includes `plugins/maister/bin/maister-install.mjs` and `plugins/maister/bin/maister-agent-gate.mjs` (`:113-127`). This is the archive contract the launcher must consume, not duplicate.

`plugins/maister/bin/release-interface.mjs:175-187` writes `plugins/maister/.maister-source.json` with `schema_version`, `source_commit`, `source_version`, and a `content_hash` calculated over the staged tree while excluding the manifest itself. `:190-211` copies and validates the E3 attestation against source commit, version, portable-core hash, and artifact digest. `:232-269` produces a sorted ustar stream with normalized ownership/timestamps, and `:272-299` writes `maister-<target>.tar.gz` and removes build staging in `finally`.

`plugins/maister/bin/release-metadata.mjs:11-17` allowlists the archive filename pattern. `:109-132` maps every `SUPPORTED_TARGET_IDS` value to `maister-${target}.tar.gz`, requires exactly the three archives and matching `SHA256SUMS`, and records per-archive size/hash/E3 observations. `:135-152` requires identical E3 bytes and source identity across targets. `:271-319` verifies the SBOM, unsigned provenance, archive hashes, and E3 properties. These functions establish release metadata invariants, but they expect a local directory containing all target archives and are not a complete one-asset remote resolver or hostile extractor.

### Release workflow

`.github/workflows/release.yml:1-19` is tag-driven (`v*`) and runs on Node 22. `:20-45` orders validation, strict parity, core/runtime tests, one source-bound E3 attestation, all three packages, and extracted lifecycle smoke. `:46-58` generates and verifies `SHA256SUMS`, `SBOM.cdx.json`, and `PROVENANCE.json`; `:59-76` uploads `dist/*` and publishes all of it to the GitHub Release.

This workflow gives the launcher a stable intended asset shape—`v<version>/maister-<target>.tar.gz` plus sidecars—but it has no npm package publication, `npm pack` inspection, public npx smoke, or npm/GitHub alignment gate. The launcher work must extend the release contract without weakening the existing clean-checkout, parity, E3, package, or lifecycle gates.

## Installer and mutation boundary

`plugins/maister/lib/distribution/cli-contract.mjs:15-18` defines the closed lifecycle command and evidence sets. `:27-52` requires a target, requires `--source` only for `install`/`update`, rejects unknown options, and limits E3 paths to mutating commands. `:55-79` maps typed failures to the established exit codes 2–8 and defines the stable JSON envelope shape.

`plugins/maister/bin/maister-install.mjs:28-90` parses the request, resolves a GitHub or local source for `install`/`update`, selects the overlay from the resolved source root, loads E3, computes the portable-core hash, binds E3 to source provenance, and calls `executeLifecycle`. `:91-119` returns the installer envelope; its `finally` already invokes resolver cleanup. The launcher should invoke this CLI as a child with non-shell argument passing rather than bypassing the public contract.

`plugins/maister/lib/distribution/transaction-manager.mjs:1221-1274` is the lifecycle authority. It resolves and revalidates a source only for `install`/`update` (`:1221-1229`), reads the active receipt for `status`/`verify` without source resolution (`:1233-1243`), then owns the lock and routes uninstall, rollback, and recovery (`:1245-1274`). The target state layout is created by `:281-293` and includes private state, journals, receipts, backups, staging, and `install.lock`.

`plugins/maister/lib/distribution/targets.mjs:5-40` is the target registry: Codex manages `.codex/plugins/local/maister`, Cursor manages `.cursor/plugins/local/maister`, and Kiro manages `.kiro-maister` plus a leaf set under `.kiro/agents`. `plugins/maister/lib/distribution/target-paths.mjs:19-47` derives those roots and state paths. The launcher must reuse this registry for target validation and must never reproduce managed-root ownership rules.

The current state-only requirement has a concrete integration consequence. The overlays show only host projection inputs: Codex `plugins/maister/overlays/codex/overlay.yml:9-34`, Cursor `.../cursor/overlay.yml:9-34`, and Kiro `.../kiro-cli/overlay.yml:9-34`. They do not install `plugins/maister/bin/maister-install.mjs` into the managed target root. Therefore a launcher that ships only its own front-end cannot delegate offline state-only commands to the archive's installer after npm installation. The implementation must either ship a version-aligned installer/control plane and overlays in the npm package, locate a separately verified control plane, or explicitly change the offline contract; this is not solved by `cli-contract.mjs` alone.

## Existing immutable-source and safety seams

- `plugins/maister/lib/distribution/source-resolver.mjs:30-75` validates safe refs and `github:owner/repo`; `:393-451` resolves a GitHub source to one full commit and a private detached checkout with cleanup; `:453-533` verifies the root, HEAD, status, content hash, source version, and cleanup binding. This is a useful identity/timeout/cleanup pattern, but it resolves Git checkouts, not GitHub Release assets.
- `plugins/maister/lib/distribution/source-resolver.mjs:265-321` recognizes the embedded `.maister-source.json`, rejects non-full commits and ref mismatches, recomputes the content hash while excluding the manifest, and returns an archive source binding. After safe extraction, the launcher should pass the extracted root through this existing archive-source path rather than inventing another manifest interpretation.
- `plugins/maister/lib/distribution/path-safety.mjs:23-57` provides relative-path normalization and containment; `:80-142` captures and revalidates path identity without following symlink components; `:183-235` provides no-follow regular-file reads. These primitives are appropriate for output-root and manifest handling, but archive entry parsing still needs a dedicated policy for tar headers and link metadata.
- `plugins/maister/lib/distribution/materializer.mjs:235-288` detects normalized destination collisions; `:1000-1128` creates same-filesystem staging, revalidates source before/after assembly and projection, checks symlinks, inventories, modes, syntax, references, hashes, content, and provenance, and removes newly created staging on failure. It is the downstream defense-in-depth boundary, not a substitute for validating archive bytes before extraction.
- `plugins/maister/lib/distribution/e3-attestation.mjs:96-145` validates the exact E3 schema, result, timestamps, hashes, and digest; `:189-214` loads a single self-contained attestation; `:223-276` binds commit, source version, portable-core hash, and artifact digest. The launcher must require a fresh `passed` attestation and leave final consumption to the installer.
- `plugins/maister/bin/maister-agent-gate.mjs:43-58` demonstrates a bounded ESM CLI with typed JSON failure output. It is part of the packaged runtime closure and is asserted by `tests/platform-independent/release-package.test.mjs:294-309`; it is not a second installer authority. Preserve it in archive closure and extracted-runtime smoke tests.
- `plugins/maister/lib/distribution/recovery.mjs:43-97` and `:580-620` provide exact-root path checks, private-tree cleanup, recovery residue removal, and durable failure handling. `plugins/maister/lib/distribution/transaction-manager.mjs:269-317` and `:377-410` show directory permissions, fsync, exclusive lock creation, and `finally`-based descriptor/lock cleanup. The launcher should reuse the same exact-root cleanup discipline and never delete installer state on cleanup failure.

## Launcher integration shape

The launcher should be a separate distribution boundary with a small set of responsibilities:

1. Parse command, explicit target, `--json`, and supported pass-through options before network or host filesystem work. Validate the target against `SUPPORTED_TARGET_IDS` and reject ambiguous release selectors early. Build the child argument vector only after release verification; do not expose arbitrary URLs or arbitrary archive filenames.
2. Resolve the npm selector to one release identity. An exact package version `X.Y.Z` must require tag `vX.Y.Z`, a published non-draft/non-prerelease release, exactly one allowlisted target asset, and matching source version. For `@latest`, use the concrete version installed by npm as the expected version, report `latest → X.Y.Z`, and reject npm/GitHub skew rather than asking GitHub's moving “latest” endpoint to choose a different archive.
3. Download only expected release metadata and the selected archive into one unique private temporary root. Enforce HTTPS/host allowlists, bounded redirects, successful status, response-size limits, stream-size limits, and cleanup registration immediately after root creation. Never log credentials, authorization headers, or full local extraction paths in machine output.
4. Verify the selected archive before extraction. Require the expected asset name, local SHA-256, matching `SHA256SUMS`/SBOM/provenance observations where present, matching GitHub asset digest where present, and coherent release tag/version/source identity. A disagreement is an error; absence of an expected binding is not permission to continue.
5. Inspect archive entries without allowing them to write. Reject NULs, absolute paths, drive-letter and UNC paths, empty/dot/dotdot segments, backslash ambiguity, duplicate and case-colliding normalized names, hard links, unsafe symlinks, devices/FIFOs/sockets, unexpected ownership or modes, excessive entry count, compressed/expanded-size limits, and unexpected top-level layout. Require exactly the expected `plugins/maister` runtime shape, selected overlay assets, contract-only sibling overlays, `.maister-source.json`, and E3.
6. Extract only into a newly created restrictive directory, with no pre-existing links and no overwrite of existing entries, and revalidate the output tree and source manifest afterward. Use a vetted cross-platform archive implementation or a non-shell subprocess with verified capabilities and equivalent pre/post checks; do not copy the trusted-archive `tar` calls from the existing tests as untrusted input handling.
7. For `install`/`update`, invoke the verified `plugins/maister/bin/maister-install.mjs` with the lifecycle command, explicit target, `--source local:<temporary-root>`, the manifest's full commit as `--ref`, and the embedded E3 path only if the installer contract needs an explicit path. Do not rewrite the manifest or attestation. The installer will resolve/revalidate the archive binding and perform its own E1–E4/materialization/transaction checks.
8. For state-only commands, use the separately resolved control-plane strategy described above and invoke the same installer CLI without release download. The command must work with GitHub unavailable and must not infer a receipt, backup, or rollback target in the launcher.
9. Forward child stdout/stderr, JSON, receipt/journal paths, exit status, and signals faithfully. Pre-delegation errors need a namespaced launcher phase/kind without colliding ambiguously with installer codes; post-delegation codes 2–8 remain authoritative. Cleanup runs in `finally` for success, verification failure, child failure, timeout, and signal; cleanup failure is a warning that must not mask the primary result or trigger host rollback.

## Constraints and acceptance invariants

- Only the exact artifact selected by the npm package version and explicit target may be downloaded or delegated; no default branch, source checkout, alternate asset, prerelease, or nearby version fallback.
- No host mutation occurs before release resolution, transport, digest, archive, source-manifest, topology, and E3 checks pass.
- The launcher writes only below its exact invocation-owned temporary root and removes only that root. It never removes target, home, state, lock, receipt, journal, backup, or staging paths.
- The installer remains responsible for target ownership, settings drift, locks, journals, receipts, materialization, integrity, rollback, and recovery. A launcher result must not claim a mutation succeeded when the child returned a nonzero or signal-terminated result.
- Stdout/JSON, stderr diagnostics, exit codes, receipt paths, journal paths, and cleanup warnings must be composable and documented.
- The implementation must reuse the existing target/source/E3/path-safety/materializer/transaction/recovery seams and must not make a direct trusted-build `tar` extraction call the untrusted-input boundary.
- Release CI must continue to build all targets from one clean commit and one E3 record, verify checksums/SBOM/provenance, smoke extracted lifecycle behavior, and publish only artifacts generated and verified in that job.

## Bottom line

The existing codebase makes the mutation and payload boundaries clear enough for an additive launcher, but the work is not just “add an npm bin.” The release resolver, safe archive verifier/extractor, child-process/signal contract, npm/GitHub publication alignment, and offline state-only control plane are new high-risk seams. Resolve the state-only control-plane choice and archive implementation/engine policy first; then build the launcher around the already-tested release archive and installer contracts, adding hostile-input and public-package tests before updating release documentation or publication automation.
