Phase 5 · MADR decision log

Thin npx release launcher decisions

Five accepted architecture records for package boundaries, release payload selection, reproducibility, archive safety, and coordinated publication.

TL;DR

@mateuszrapacz/maister will be a thin distribution launcher. GitHub Release archives remain authoritative payloads, exact versions remain reproducible, latest resolves visibly, all verification fails closed before delegation, and npm/GitHub outputs are promoted as one aligned release.

maister-install.mjs remains the only authority that writes host files, settings, receipts, journals, backups, locks, or recovery state.

Key Decisions

  1. Thin npm launcher and one mutation authority.
  2. GitHub Release payloads and target allowlist.
  3. Exact version baseline and visible latest resolution.
  4. Fail-closed verification and safe extraction.
  5. Coordinated publication and provenance handling.

Open Questions / Risks

  • The minimum Node engine is measured during implementation; Node 22 is the current CI reference.
  • The extraction implementation must prove the accepted cross-platform archive policy.
  • Unsigned integrity records do not authenticate the publisher; trusted publishing/signing is a follow-up ADR.
  • npm/GitHub publication is not atomic and requires promotion, retry, deprecation, and skew-response procedures.
Decisions: ADR-001 · ADR-002 · ADR-003 · ADR-004 · ADR-005 · Summary
Accepted

ADR-001: Thin npm launcher and one mutation authority

Context

The existing installer already owns source resolution, overlays, E3, staging, locks, journals, managed settings, receipts, drift, rollback and recovery. The missing capability is one-command public distribution, not a second lifecycle implementation. Bundling mutation logic into npm or generic skill-copy tooling would split authority and guarantees.

Decision Drivers

  • One owner for host state and lifecycle invariants.
  • Reuse tested transactional behavior and state schemas.
  • Small npm package and minimal semantic drift.
  • Offline state-only lifecycle commands.

Considered Options

  1. Complete npm payload plus new mutation logic.
  2. Generic Agent Skills copy/symlink tooling.
  3. Thin verified release launcher delegating to the existing installer.
  4. Host marketplaces first.

Decision Outcome

Choose option 3. The launcher owns parse, resolve, download, verify, safe temporary extraction, delegation, forwarding and cleanup. Only maister-install.mjs crosses into host and durable installer state.

Consequences

Good
  • Manual archive and npx paths share transaction semantics.
  • Existing receipts remain authoritative.
  • Pre-delegation failures are provably mutation-free.
Bad
  • Node/npm is required for this channel.
  • Launcher/installer CLI compatibility needs fixtures.
  • The release archive still carries installer/runtime content.

Accepted

ADR-002: GitHub Release payloads and target allowlist

Context

The release pipeline already creates deterministic self-contained archives with source manifest, selected overlay, installer, and E3. Duplicating those payloads in npm would establish a second artifact source. Arbitrary URLs or auto-detected targets would weaken release and host identity.

codex    → maister-codex.tar.gz
cursor   → maister-cursor.tar.gz
kiro-cli → maister-kiro-cli.tar.gz

Decision Drivers

  • One authoritative target payload per version.
  • Reuse deterministic package tests.
  • Explicit host choice.
  • Reject repository, URL, asset and cross-target substitution.

Considered Options

  1. Bundle all payloads in npm.
  2. Build from default branch.
  3. Accept arbitrary archive URLs.
  4. Resolve fixed target assets from matching repository releases.

Decision Outcome

Choose option 4. Resolve only mateuszrapacz/maister releases. Require an explicit codex, cursor, or kiro-cli target and select exactly one closed-map asset. npm carries launcher code only.

Consequences

Good
  • Existing release artifacts become the npx contract.
  • Cross-target selection fails before extraction.
  • npm stays small.
Bad
  • Install/update depends on npm and GitHub availability.
  • Rate limits, redirects and CDN behavior enter scope.
  • Release asset retention becomes operational.

Accepted

ADR-003: Exact version baseline and visible latest resolution

Context

An exact npm version is reproducible; latest is a moving dist-tag resolved before the binary runs. Independently asking GitHub for its latest release can select a different version during propagation or partial publication. Silent moving selection makes audits and incident diagnosis ambiguous.

Decision Drivers

  • Replayable CI and runbook commands.
  • Audit from selector to source commit.
  • npm/GitHub skew detection.
  • Convenient first-install command without hidden movement.

Considered Options

  1. Only runtime GitHub latest.
  2. Exact versions only.
  3. Exact plus latest, where the concrete npm package version must match GitHub and is shown.
  4. A separate overriding version flag.

Decision Outcome

Choose option 3. Package @X.Y.Z resolves only tag vX.Y.Z. For @latest, npm's concrete package version becomes the only acceptable GitHub release. Output includes requested and resolved versions before delegation; the receipt binds resolved source version, full commit and content hash.

Consequences

Good
  • Exact commands are replayable.
  • Latest remains easy but observable.
  • Channel skew fails closed.
Bad
  • Docs must distinguish convenience and reproducibility.
  • Dist-tag propagation can vary across caches.
  • Durably storing the literal selector would need a receipt-schema ADR.

Accepted

ADR-004: Fail-closed verification and safe extraction

Context

The launcher introduces network bytes and archive extraction before the installer can apply its own provenance and transaction checks. Digest-only validation does not stop traversal, link escape, special files, decompression bombs, unexpected topology, or a correctly hashed but wrongly bound payload.

Decision Drivers

  • Zero host mutation for release/archive failures.
  • Defense in depth across transport, digest, archive, source and E3.
  • Cross-platform path safety and bounded resources.
  • Deterministic cleanup while preserving installer authority.

Considered Options

  1. Default tar extraction and trust the installer.
  2. Digest-only verification.
  3. Inspect, verify, extract privately, revalidate, then delegate local source.
  4. Stream directly into host targets.

Decision Outcome

Choose option 3. Validate release, expected asset, bounded download, computed digest, archive paths/types/links/counts/sizes/topology, source manifest and fresh passed E3 before delegation. Extract only to an exclusive private temp root and revalidate the tree. Cleanup deletes only that exact root.

Consequences

Good
  • Archive bytes cannot directly reach host state.
  • Pre-delegation failure has an unchanged-state invariant.
  • Release skew is visible through layered bindings.
Bad
  • Archive safety is the largest new test surface.
  • Inspection/extraction uses extra I/O and disk.
  • Topology policy must evolve with package format.

Accepted

ADR-005: Coordinated publication and provenance handling

Context

The public path spans npm launcher version X.Y.Z and GitHub tag vX.Y.Z with three archives and sidecars. These channels cannot update atomically. Current SHA256SUMS, SBOM, provenance and E3 create consistency evidence but are unsigned and are not independent publisher authentication.

Decision Drivers

  • One source version/commit across all outputs.
  • No latest promotion before public end-to-end proof.
  • Immutable artifacts and idempotent retries.
  • Accurate integrity-versus-authentication claims.

Considered Options

  1. Publish independently and align manually.
  2. Publish npm first and wait for GitHub.
  3. Publish/verify GitHub, publish exact npm, smoke exact npx, then promote latest.
  4. Block distribution until a signing system exists.

Decision Outcome

Choose option 3. Validate, package and publish GitHub assets first; verify public assets; pack/smoke and publish the matching exact npm version; run a public exact npx smoke; only then move latest. Retries reuse identical bytes. Broken releases are withheld/deprecated and fixed with a new patch.

Consequences

Good
  • Stable latest represents a proven public chain.
  • Version/source/E3 skew is gated.
  • Partial publication has a recovery policy.
  • Trust claims remain accurate.
Bad
  • Release automation is longer and cross-service.
  • Exact versions may precede latest promotion.
  • Maintainers need protected, retryable publication procedures.
