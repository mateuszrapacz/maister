Phase 5 · High-level design

Thin npx release launcher

A verified distribution adapter for @mateuszrapacz/maister that resolves GitHub Release payloads and preserves maister-install.mjs as the only host mutation authority.

TL;DR

Publish a small Node.js ESM package with a maister binary. For install/update it binds the concrete npm package version to a matching mateuszrapacz/maister GitHub Release, selects one allowlisted target archive, verifies digest, topology, source manifest and E3, extracts privately, and delegates to the archive's installer.

The launcher writes only invocation-owned temporary data. Existing installer locks, journals, transactions, settings ownership, receipts, rollback and recovery remain unchanged. Exact versions are reproducible; latest must show the exact package/release selected.

Key Decisions

  • Thin launcher and one mutation authority — ADR-001.
  • GitHub Release payloads with a closed target map — ADR-002.
  • Exact baseline plus visible latest resolution — ADR-003.
  • Fail-closed verification and safe private extraction — ADR-004.
  • Coordinated npm/GitHub publication with honest provenance claims — ADR-005.

Open Questions / Risks

  • EngineThe Node floor must be measured during implementation; Node 22 is only today's release-CI reference.
  • TrustUnsigned checksums and provenance detect inconsistency but do not independently authenticate the publisher.
  • ExtractionThe safety policy is fixed; the cross-platform library/tool remains to be proven against adversarial fixtures.
  • Releasenpm and GitHub publication is not atomic, so promotion and partial-failure handling are explicit gates.

Architecture style and boundaries

User / CI
   | npx @mateuszrapacz/maister@<selector> <command> --target <target>
   v
+--------------- npm distribution adapter ----------------+
| parse | resolve | download | verify | inspect | extract |
|                    delegate | forward | cleanup          |
+----------------------------|------------------------------+
                             | local:<verified-temp-root>
                             v
+--------------- existing mutation authority --------------+
| maister-install.mjs                                      |
| E3 | materialize | lock | journal | transaction | receipt|
| drift | rollback | recover                               |
+----------------------------|------------------------------+
                             v
                   managed host state

Distribution boundary: turns a selector into a verified local payload and may write only its private temporary root. Transaction boundary: turns that payload into host state and is crossed only by maister-install.mjs.

Text fallback/description: the flow is linear from npx to a verified local extraction, then into the existing installer, then into host state. No launcher edge directly reaches host state.

Lifecycle flows

Install / update

parse → exact release → allowlisted asset → private temp → bounded download → digest → inspect → extract → manifest + E3 → show provenance → delegate → forward → cleanup

Every pre-delegation failure leaves host and installer state unchanged. Once delegation starts, only the installer may compensate or recover.

Status / verify

parse target → delegate locally → forward receipt/drift result

No network or release resolution. Status reads installed state; verify applies existing receipt, inventory, settings and integrity checks.

Uninstall / rollback

parse target → delegate → installer journal/ownership checks → forward terminal result

The launcher never pre-deletes host files, settings, receipts, backups or locks.

Recover

operator quiesces host → delegate offline → preserve journal/backups → verify receipt → forward

Code 7 remains unresolved state. The launcher does not recover by deleting locks or retrying blindly.

Exact and latest resolution

Exact @X.Y.Z

  1. Require GitHub tag vX.Y.Z.
  2. Require exactly one mapped target asset.
  3. Require release, manifest and E3 source version X.Y.Z.
  4. Bind full commit and content hashes through final receipt.
  5. Never fall back.

Moving @latest

npm resolves the dist-tag before execution. The running package's concrete version becomes the only acceptable GitHub version. Do not independently choose GitHub's moving “latest release.”

Resolved latest -> 2.2.1 (v2.2.1)
target codex · asset maister-codex.tar.gz

JSON carries requested and resolved values; the receipt durably binds resolved source version and commit.

Verification, extraction and cleanup

  1. Inputs: strict command/version and closed repository/target/asset policy.
  2. Transport: HTTPS, expected hosts, bounded redirects, timeout, status and streamed bytes.
  3. Release: published stable release, exact version/tag, unique expected asset.
  4. Digest: locally computed SHA-256 agrees with selected release records and GitHub digest when available.
  5. Archive inspection: reject absolute, drive, UNC, traversal, ambiguous, duplicate/case-colliding paths; unsafe links, special files, modes, counts and sizes.
  6. Private extraction: exclusive restrictive temp root; no overwrite or path escape.
  7. Post-extraction topology: exactly one known installer, selected overlay, source manifest and E3; no cross-target payload.
  8. Source + evidence: version, full commit, content/core/artifact hashes and fresh passed E3 all agree.

Cleanup invariant

Register cleanup immediately, execute it in finally, and delete only the exact invocation-owned temp root. Cleanup failure preserves the primary result and reports the retained path; it never triggers host rollback.

Error, JSON, signal and exit-code forwarding

Before delegation, errors name the launcher phase and resolved release context known so far. After delegation, installer stdout, stderr, JSON, receipt/journal paths and exit status are authoritative and remain unchanged.

CodeExisting meaningLauncher behavior
0SuccessForward with release context.
2Usage/settings formatNo retry.
3Source resolutionNo version or branch fallback.
4Validation/evidenceNo alternate asset.
5DriftOperator decision required.
6Lock busyNo blind retry loop.
7Transaction/recoveryPreserve and report durable state.
8IntegrityStop and investigate.

With --json, stdout is machine-readable only. Signals are forwarded to the child; interruption cannot become success.

npm and GitHub publishing alignment

clean vX.Y.Z → validate/parity/E3/package → deterministic 3-target archives → verify SHA256SUMS/SBOM/PROVENANCE → publish GitHub → verify public assets → npm pack/smoke → publish exact npm X.Y.Z → public exact npx smoke → move latest

  • npm version, GitHub tag, source manifest/version/commit and E3 bytes must align.
  • Do not move latest before the exact public chain passes.
  • Retries reuse identical immutable bytes; fixes use a new patch version.
  • A partial/broken release is withheld from promotion and deprecated where appropriate.

Security, trust and observability

Threats addressed

  • Wrong target/asset: allowlist.
  • Version substitution: npm/tag/manifest/E3/receipt binding.
  • Corruption: local digest and release observations.
  • Archive attacks: bounded inspect-before-extract.
  • Partial host writes: existing transaction/recovery.
  • Moving input: visible latest resolution.

Residual trust risk

Current sidecars and E3 are not an independent publisher signature. A compromise that produces internally consistent artifacts remains possible. Trusted npm publishing and stronger signing/attestation are a separate decision.

Local observability

resolve → download → verify_digest → inspect → extract → verify_source → delegate → cleanup

Human and JSON output report run ID, selector, exact version/tag, target, asset, digest, source commit, result code and receipt/journal paths. Credentials are redacted. No remote telemetry is required.

Compatibility and test strategy

Compatibility

  • Explicit Codex, Cursor and Kiro CLI only.
  • Linux, macOS and Windows path/link/temp/signal coverage.
  • Engine floor declared only after Node/npm matrix verification.
  • Delegate to archive installer of the exact same version.
  • No launcher-owned persistent state schema.

Test layers

  • CLI/resolver/allowlist/JSON/forwarding units.
  • Network status, redirects, timeout and size limits.
  • Malicious archive path/link/type/count/size corpus.
  • Manifest/source/E3 mismatch and freshness tests.
  • Three-target lifecycle with isolated home/state.
  • Drift, lock, transaction, rollback and recovery forwarding.
  • npm pack, release alignment and exact public npx smoke.

Critical rejection invariant: every pre-delegation failure leaves target, managed settings and installer state absent or byte-exact unchanged.

Rollout

  1. Freeze CLI, target map, release descriptor, error/JSON fixtures and malicious archive corpus; verify Node floor.
  2. Build and test a local packed launcher with mocked GitHub evidence.
  3. Run three-target lifecycle in isolated environments on supported operating systems.
  4. Add version/commit/E3 release alignment and publication-order gates.
  5. Publish matching GitHub assets and first exact npm version; smoke exact public npx.
  6. Promote latest; rollback the dist-tag, not installed host state, if the channel fails.
  7. Evaluate trusted publishing/signing in a follow-up ADR.

Success boundary

The npx channel is complete when exact/latest resolution is auditable, all verification failures are mutation-free, all seven lifecycle outcomes forward unchanged, all three targets pass isolated end-to-end tests, temporary data is bounded and cleaned, and npm/GitHub versions are gated as one release unit.
