Solution exploration

Maister release distribution

How to provide one-command installation while preserving Maister's transactional, multi-host lifecycle.

TL;DR

Ship a thin public @mateuszrapacz/maister launcher now. It verifies the matching mateuszrapacz/maister release archive and delegates all host mutation to maister-install.mjs. Add host-native marketplaces later.

Key Decisions

  • GitHub Releases hold payloads.
  • npm supplies the entry point.
  • The existing installer remains sole lifecycle authority.
  • Targets and reproducible versions stay explicit.

Open Questions / Risks

  • Publisher authentication.
  • Safe cross-platform extraction.
  • npm/release version alignment.
  • Marketplace parity and ownership.

Alternatives

In scope · Recommended now

1. Thin npx release launcher

Resolve an exact or latest release, map the explicit target, verify and extract privately, then delegate to maister-install.mjs.

Pros: one command, small surface, exact pinning, existing guarantees preserved.

Cons: requires npm/Node and safe download/extraction logic.

Risks: version skew, unsafe archives, hidden moving versions. Mitigate with allowlists, bounded extraction, alignment tests, and visible provenance.

Stretch for skills · Primary installer out of scope

2. Agent Skills copy/symlink package

Expose standalone Maister skills through generic project/global copy or symlink workflows.

Pros: familiar, discoverable, editable, generic update tools.

Cons: does not model target overlays, settings, receipts, rollback, or recovery.

Risks: drift and confusion between “skills installed” and “Maister installed.”

Stretch · Later channel

3. Per-host managed marketplaces

Publish separately through each host's native plugin channel.

Pros: native discovery, trust, updates, uninstall, and managed read-only bundles.

Cons: duplicated packaging, review, metadata, and compatibility work.

Risks: release lag and behavioral divergence across hosts.

Stretch after launcher validation

4. Universal bootstrap plus host adapters

Use one bootstrap that dispatches to a certified native adapter when available, with the verified archive path as fallback.

Pros: one documented command and gradual native-channel adoption.

Cons: more branching, contracts, and compatibility tests.

Risks: conflicting state or lifecycle ownership between paths.

SCAMPER ideas

  • Substitute: npx for manual download, not for the installer.
  • Combine: npm execution, GitHub payloads, transactional mutation.
  • Adapt: source pinning and tracking to Maister receipts.
  • Modify: persist the version resolved from latest.
  • Put to use: a clearly labeled “skills only” catalog.
  • Eliminate: mutation logic in distribution adapters.
  • Reverse: adapters call Maister's contract, rather than Maister mimicking every host.

Recommendation

Implement the thin @mateuszrapacz/maister launcher first.

@mateuszrapacz/maister
  → mateuszrapacz/maister GitHub Release
  → maister-<target>.tar.gz
  → maister-install.mjs
  → host state

Limit the launcher to resolution, verified download, safe temporary extraction, delegation, cleanup, and result forwarding. Add host-native marketplaces later.

Why generic npx skills is insufficient: it installs skills by copy or symlink. Maister's full plugin lifecycle also owns target overlays, managed settings, staging, receipts, journals, drift detection, verification, update, uninstall, rollback, and recovery. A skills channel can complement that lifecycle, but cannot replace it without losing or duplicating guarantees.

Deferred ideas

  • Host-native marketplace packages.
  • Certified host adapters behind a universal bootstrap.
  • A catalog entry for standalone Maister skills.
  • Stronger publisher authentication or trusted publishing.
  • Advisory host auto-detection while explicit --target remains authoritative.
