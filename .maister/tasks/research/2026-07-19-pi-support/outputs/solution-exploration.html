TL;DR

Five material decisions were explored through HMW/SCAMPER and evaluated for technical feasibility, user impact, simplicity, risk, and scalability. The strongest coherent solution is a private generated Pi package, hybrid prompt/skill projection, a thin pi.native adapter over delegation v1, identity-aware ownership of one package-array entry, and greenfield admission beside frozen historical parity. Maister owns canonical roles, resolution, policy, durable events, receipts, and evidence; Pi loads resources and pi-subagents executes children. Support stays version-bound and graduated through E1–E6.

Key Decisions

  • One deterministic private package with an explicit Pi manifest.
  • Skills for workflows, prompts for textual commands, extension only for runtime bridging.
  • Delegation v1 owns dispatch only; Maister owns resolution and durable events.
  • Manage one normalized packages[] member and preserve all other Pi state.
  • Historical three-target oracle remains frozen; all-target greenfield gates admit Pi.

Open Questions / Risks

  • Cancellation and resumed inspection are not complete public delegation-v1 capabilities.
  • Exact prompt-versus-skill mapping still needs per-command review.
  • RPC source metadata is version-sensitive.
  • Array-entry normalization and rollback need strong negative testing.

Method and evaluation frame

HMW questions asked how to become Pi-native without a second source, keep commands native without turning them all into code, delegate exact agents without surrendering auditability, own one package without owning user state, and admit Pi without rewriting historical parity. SCAMPER generated substitutes, combinations, adaptations, eliminations, and reversals. Scores are 1–5: T feasibility, U user impact, S simplicity, R risk posture, Sc scalability.

1. Package and discovery integration

This boundary decides whether Pi support is deterministic and receipted or dispersed across mutable user directories.

Recommended · 1A

One private generated Pi package

Materialize a complete target package under a Maister-private root with explicit manifest entries for extensions, skills, and prompts. Pi discovers one local package identity while Maister owns its bytes and lifecycle.

Pros: atomic ownership; deterministic hashes/archive; native source metadata; clean uninstall; no behavior duplication.

Cons: requires package-array ownership and Pi projector; local source is less conventional than npm.

Evidence: isolated package probe, distribution finding 04, canonical materialization and receipts.

Assumptions: local paths and explicit manifests remain supported; private root stays trusted.

T5 · U4 · S4 · R5 · Sc5 · confidence high

1B. Loose global resources

Install directly into global extension, skill, and prompt directories. Discovery is immediate, but ownership is fragmented across operator-shared locations.

Pros: simple mental model; minimal package machinery; direct reload.

Cons: collisions; fragmented receipts; difficult drift/uninstall; shared mutations.

Evidence: supported by Pi, contrary to Maister's contained whole-tree preference.

Assumptions: leaf ownership can be race-safe across three roots.

T4 · U3 · S2 · R2 · Sc2 · confidence high

1C. Published npm package with bundled subagents

Ship Maister and subagents as one npm unit managed by pi install. This improves conventional UX but expands dependency and supply-chain ownership.

Pros: familiar installation; one external version; portable.

Cons: duplicates operator extension; external publishing; broader rollback and peer-resolution risk.

Evidence: npm is supported, but dry-run analysis showed material duplication.

Assumptions: publishing and offline reproducibility are controllable.

T3 · U5 · S3 · R2 · Sc4 · confidence medium

Recommendation: 1A uniquely combines canonical ownership, deterministic releases, native discovery, and transactional uninstall.

2. Command and skill projection

Pi has no native commands directory; slash behavior comes from prompts, skills, or extension registration.

Recommended · 2A

Hybrid semantic projection

Project reusable workflows as skills, textual command expansions as prompt templates, and imperative host actions as extension commands. A closed mapping gives each canonical command one representation and verifies origin through RPC.

Pros: faithful semantics; minimal code; progressive disclosure; testable origins.

Cons: three transforms; mapping review; mixed slash naming.

Evidence: Pi resource contract and Maister semantic bindings directly support it.

Assumptions: every command can be classified without dual registration.

T5 · U4 · S4 · R5 · Sc5 · confidence high / exact mapping medium-high

2B. Everything as prompt templates

Render all commands as Markdown prompt templates. Invocation is uniform, but complex workflows and assets collapse into expanded prompt text.

Pros: simple files; uniform slash UX; minimal extension.

Cons: loses skill assets/progressive disclosure; large prompts; semantic drift.

Evidence: prompts are native, but cannot faithfully replace structured skills.

Assumptions: behavior is mostly textual.

T4 · U5 · S5 · R3 · Sc3 · confidence medium

2C. Everything as extension commands

Register every command from one TypeScript extension and load canonical content programmatically. Exact UX comes at the cost of converting declarative behavior into trusted code.

Pros: exact names; centralized collisions; arbitrary behavior.

Cons: large code surface; behavior-bearing projection; harder parity.

Evidence: registerCommand supports it, but minimal/single-source principles oppose it.

Assumptions: generated code stays safely equivalent.

T4 · U5 · S2 · R2 · Sc3 · confidence high that inferior

Recommendation: 2A with a closed versioned mapping and fail-fast duplicate/collision checks.

3. Native bridge shape

The bridge must execute exact native children without reimplementing subagents or surrendering identity, policy, and evidence.

Recommended · 3A

Thin direct-import delegation adapter

pi.native wraps public delegation v1: Maister resolves the role and records durable intent before calling delegate() with bounded task, cwd, and agent. Returned observations map into Maister's stream; absent cancellation/resume is explicit.

Pros: public API; exact selected agent; structured trail; smallest bridge.

Cons: process-local; cancel/resume gaps; module/peer resolution integration.

Evidence: installed successful delegation probe and public export.

Assumptions: delegation v1 remains supported for the pinned tuple.

T5 · U4 · S5 · R4 · Sc4 · confidence high dispatch / medium integration

3B. Parent-facing tool/RPC mediation

Invoke subagents through Pi's parent tool or RPC prompt and infer completion from parent events. It fits Pi sessions but inserts model/command mediation.

Pros: natural host lifecycle; no direct import; active-session UI.

Cons: lost identity/result detail; accepted prompt is not completion; model-dependent routing.

Evidence: foreground launch works but loses deterministic detail.

Assumptions: future RPC exposes stable child terminal events.

T3 · U5 · S3 · R2 · Sc3 · confidence medium-low

3C. Maister-owned reimplementation

Copy internal subagent logic and own child processes, concurrency, cancellation, and transport. Control improves, but native extension reuse disappears.

Pros: exact port fit; custom durability and resume.

Cons: duplication; unstable internals; large security/maintenance surface; violates direction.

Evidence: directly conflicts with public-boundary conclusions.

Assumptions: internals can be tracked indefinitely.

T2 · U3 · S1 · R1 · Sc2 · confidence high reject

Recommendation: 3A. Maister owns resolution, validation, durable before/after events, cancellation-request semantics, and evidence; delegation owns only what it observably executes.

4. Settings and package ownership

Pi's packages setting is shared; Maister currently owns whole keys, not one identity-normalized member.

Recommended · 4A

Identity-aware managed array entries

Add a closed managed_array_entries rule for one normalized package identity. Lifecycle preserves ordering and byte-exact unrelated entries while detecting owned-member drift and aliases.

Pros: narrow ownership; transactional rollback; reusable; detects aliases.

Cons: new primitive; subtle normalization/order; many rejection tests.

Evidence: directly fills finding 04's transaction-model gap and Pi dedupe rules.

Assumptions: local/npm/git identity normalization can be closed and versioned.

T4 · U5 · S4 · R4 · Sc5 · confidence high

4B. Own the complete packages key

Snapshot and merge the whole package array with existing managed-key machinery. Implementation is easy, but Maister becomes authoritative over operator packages.

Pros: minimal schema work; simple restore.

Cons: broad ownership; concurrent edits conflict; rollback may overwrite user intent.

Evidence: current transaction can do it, architecture forbids this ownership expansion.

Assumptions: no external package edits occur.

T5 · U2 · S4 · R1 · Sc2 · confidence high unsafe

4C. Manual operator registration

Install private bytes but require the operator to add/remove the settings entry. Shared-file risk disappears, while atomic installation and discoverability proof disappear too.

Pros: no shared settings mutation; minimal installer change.

Cons: not first-class; manual errors; ambiguous E4/E5; poor uninstall.

Evidence: safe but below requested lifecycle guarantees.

Assumptions: manual maintenance is acceptable.

T5 · U1 · S3 · R4 · Sc2 · confidence high

Recommendation: 4A with fail-before-mutation tests for aliases, normalized collisions, invalid shapes, drift, symlinks, and concurrency.

5. Greenfield admission and release

The historical parity oracle records a three-target migration boundary; retroactively inserting Pi would destroy that meaning.

Recommended · 5A

Historical parity plus greenfield admission

Freeze the historical oracle and admit Pi through all-target registry, projection, lifecycle, evidence, topology, and release matrices. E1-E4 authorize only structural/installable packaging; fresh E5/E6 authorize native/semantic claims.

Pros: preserves history; clean architecture; graduated claims; deterministic release.

Cons: two gate families; documentation burden; larger CI matrix.

Evidence: matches synthesis, roadmap policy, and compatibility finding.

Assumptions: release can require both gate families.

T5 · U4 · S4 · R5 · Sc5 · confidence high

5B. Expand historical oracle to Pi

Add Pi to the immutable legacy comparison and require historical equivalence. This creates one matrix by fabricating a baseline that never existed.

Pros: one headline parity gate.

Cons: corrupts oracle; fabricated history; encourages generated legacy artifacts.

Evidence: contradicts architecture migration boundary.

Assumptions: a defensible historical source exists.

T2 · U3 · S3 · R1 · Sc2 · confidence high reject

5C. Experimental side-channel release

Publish a preview package/manual path outside the main matrix and defer full evidence integration. Feedback arrives sooner, but a second distribution architecture can become permanent.

Pros: fastest feedback; isolates instability.

Cons: weak provenance; inconsistent UX; experimental debt; ambiguous claims.

Evidence: provisional packaging still requires the normal E1-E4 architecture.

Assumptions: preview is strictly time-boxed.

T4 · U3 · S2 · R2 · Sc2 · confidence medium

Recommendation: 5A. Keep registered, structural, installable, discovery-verified, runtime-verified, and semantically verified claims distinct.

Implementation sequence

  1. Admit Pi in closed registries and define overlay/projection.
  2. Generate skills, prompts, all 28 agents, and manifest deterministically.
  3. Add managed array entries and lifecycle fixtures.
  4. Add pi.native over delegation v1 with honest capability limits.
  5. Add E1-E6 scenarios, archive smoke, and greenfield release gates.

Deferred ideas

  • Npm marketplace publication: after local/archive/GitHub lifecycle stabilizes.
  • Vendored subagents: only if operator-owned prerequisite proves inadequate.
  • Full background resume/cancel: when public APIs expose durable IDs and cancellation.
  • Generic multi-host array ownership: reuse later, but broad adoption is separate.
  • Cross-harness skill federation: valuable ecosystem work, outside this target implementation.

Evidence and confidence

Alternatives derive from the synthesis, research report, all five findings, and project vision, architecture, roadmap, and stack. Confidence is high for Pi 0.80.10 resource/RPC behavior, subagents 0.35.1 delegation, canonical projection/transaction/evidence boundaries, and deterministic release requirements. Exact command mapping and operational import/cancellation mechanics remain medium confidence.
