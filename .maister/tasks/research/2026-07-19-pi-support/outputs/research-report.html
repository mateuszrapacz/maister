TL;DR

Maister should add pi as a fourth first-class target delivered as one generated local Pi package, with all behavior still owned by canonical Maister sources.

Native agents should run through public pi-subagents/delegation v1; Maister retains exact role resolution, append-before-side-effect recording, terminal identity verification, and its durable hash-chained event stream.

The install owns only ~/.pi/agent/maister/** plus one identity-managed packages member. Active Pi, subagents, settings, auth, sessions, trust, and other packages remain operator-owned.

The verified tuple is Pi 0.80.10, Node 25.9.0, and pi-subagents 0.35.1. Research proves feasibility; product support remains unavailable until E1–E6 implementation and evidence.

Key Decisions

  • Register pi as a real target with adapter pi.native; do not model it as a Codex/Kiro alias or an unmanaged companion install.
  • Materialize one deterministic Pi package at ~/.pi/agent/maister/, generated from canonical skills, commands, common runtime, and all 28 canonical agents.
  • Use pi-subagents/delegation protocol v1 for native foreground execution. Keep resolveAgent and readExecutionEventStream owned by Maister.
  • Add identity-aware managed_array_entries ownership for exactly one local package declaration; never own the whole packages array and never call pi install inside the Maister transaction.
  • Keep pi-subagents external and operator-owned; verify exact configured identity/version and fail closed when absent, ambiguous, incompatible, or inactive.
  • Keep the immutable historical parity oracle scoped to Codex, Cursor, and Kiro CLI; admit Pi through a separate greenfield all-target gate.
  • Make support claims graduated and evidence-driven: registered → structural → transactional → native discovery → native runtime → semantic.

Open Questions / Risks

  • The future package must prove discovery of all 28 namespaced descriptors without ambient shadowing.
  • Commands require a reviewed prompt/skill/extension mapping; Pi has no standalone commands/.
  • Foreground delegation has no public durable replay or same-run resume.
  • managed_array_entries is a new receipt/drift/rollback primitive.
  • Independent Pi updates must invalidate stale E5/E6.
  • Current historical parity validation has an unrelated exact-native.mjs baseline mismatch.

5. Native runtime adapter

Verified probe: public delegation v1 produced started, bounded updates, then completed with agent=researcher, model, run ID, exit 0 and NATIVE_API_OK. Unknown agent returned typed failed; queued cancel returned cancelled. F3 P2–P4.
  1. Maister resolveAgent validates target paths, receipt/projection/provenance, logical role, policies and capabilities; produces pi.native plan.
  2. Native port resolves active Pi/Node/subagents/protocol/auth/readiness and exact agent discovery; missing capability is unavailable.
  3. Durably append dispatch/attempt before request emission; failed append prevents launch.
  4. Subscribe before emit; use requestId=dispatch_id, exact agent, cwd/context and supported bounds.
  5. Sanitize and bound started/update fields into the Maister event schema.
  6. Map terminal status; require exact observed identity even when process/text look successful.
  7. Durably append terminal result; readExecutionEventStream reads only Maister's hash chain.
  8. On cancel or post-launch record failure emit v1 cancel and record its observable outcome.

Use

Exported pi-subagents/delegation v1 for foreground dispatch; Pi RPC for host/resource inventory.

Do not use

Parent model tool mediation, deep imports, private async status/events files, or process-local background snapshots as durable history.

Limits: no public foreground replay/same-run resume; lost execution becomes typed failure and retry is a new dispatch. Background-work exposes only process-local provider/id/session. F1 §1.6; F3 API and capability matrix.

managed_array_entries

Own exactly the package identity resolving to ~/.pi/agent/maister. Normalize relative paths as Pi does; string "./maister" and object {source:"./maister",...} are the same identity. Duplicate/conflicting identity or owned-member drift fails before mutation. Receipt records array path, normalized identity, exact entry, merge version, hashes/modes and backup. F4 §4.2. High confidence.

Lifecycle

  1. Resolve/revalidate one immutable local/archive/GitHub source and its Pi overlay.
  2. Read-only probe active Pi and configured subagents tuple.
  3. Materialize complete package and 28 roles in empty same-filesystem staging.
  4. Reject unmanaged root, symlink/path or settings identity collision, unsafe manifest, incompatible prerequisite.
  5. Snapshot roots, exact settings bytes/mode, active receipt and recovery state.
  6. Atomically replace tree, then merge one package member.
  7. Verify receipt bytes/modes/links/member; produce E4 and optional E5/E6.
  8. Publish receipt only after integrity succeeds.

Update repeats drift checks. Uninstall removes only the exact member and tree. Failure restores exact settings bytes/mode, tree topology and prior receipt. Never invoke pi install inside the transaction. F4 §§1.2, 4.3.

7. Greenfield admission vs historical parity

Historical oracle

MIGRATED_PARITY_TARGET_IDS: Codex, Cursor, Kiro CLI. Immutable pre-migration comparison stays unchanged.

Greenfield admission

SUPPORTED_TARGET_IDS: all four targets. Overlay, projection, install, runtime, evidence, archive, lifecycle, topology and release gates include Pi.

A synthetic Pi legacy tree would be dishonest. Existing targets pass both gates; Pi passes all-target admission only. F1 historical parity; F4 §6. High confidence.
