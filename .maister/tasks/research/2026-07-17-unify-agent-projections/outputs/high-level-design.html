TL;DR

Maister przyjmie architekturę portable core + ports and adapters: 28 plików plugins/maister/agents/*.md pozostanie jedynym źródłem zachowania, a wspólny parser utworzy wersjonowany manifest. Projector wygeneruje reprezentacje hostów w stagingu, przed walidacją, hashowaniem i receipt. Workflow wybierze dokładny maister:<role_id>; resolver i adapter zakończą operację błędem przy braku, kolizji, niedostępności lub mismatchu digestu.

Advisor jest zwykłą rolą. Stare pliki Codex TOML zostaną usunięte ze źródła rozwiązania, bez migracji i backward compatibility.

Key Decisions

  • Jedno źródło zachowania i wersjonowany manifest — ADR-001.
  • Deterministyczna projekcja w stagingu, związana z receipt/provenance — ADR-002.
  • Exact fail-closed resolver i natywne adaptery bez fallbacku — ADR-003.
  • Wąski receipt-owned leaf set pod natywnym rootem Kiro — ADR-004.
  • Clean install bez legacy TOML migration — ADR-005.
  • Oddzielne dowody materializacji, discovery i invocation — ADR-006.

Open Questions / Risks

  • host contract Codex nie gwarantuje stabilnego spawn schema ani per-spawn model override.
  • precedence Cursor nie dokumentuje precedence plugin-agenta wobec projektowego lub użytkownika o tej samej nazwie.
  • identity Kiro exact identity wymaga obserwowalnego, wersjonowanego probe.
  • transaction Multi-root receipt wymaga pełnych rollback, race detection i unmanaged-file preservation tests.

Design Overview

Maister ma dostarczać ten sam zestaw wyspecjalizowanych ról na Codex, Cursor i Kiro CLI bez ręcznie synchronizowanych kopii oraz bez pozornej zgodności opartej tylko na liczbie plików. Zmiana usuwa drift zachowania, naprawia brakującą rolę Cursor i niedomknięte referencje Kiro oraz daje workflow jednoznaczny kontrakt wywołania.

Wybrano warstwowy portable core z adapterami hostów, deterministyczną projekcję build/install-time i transakcyjny filesystem delivery. Format hosta może się różnić, lecz identity, digest, semantyka błędu i evidence pozostają wspólne.

  • role_id = filename stem = frontmatter.name; workflow ID: maister:<role_id>; host ID: maister-<role_id>.
  • Wszystkie 28 ról, w tym advisor, używa tej samej ścieżki.
  • explore i Kiro maister pozostają oddzielnymi support agents.
  • Generowane reprezentacje są efemerycznym wynikiem stagingu i release package.
  • Identity, closure, collision, availability i digest failures zatrzymują operację.

Data Flow

Build, install i receipt

immutable source
→ parse 28 roles
→ validate identity + manifest
→ project into empty staging
→ validate parity, syntax, refs, modes, hashes
→ calculate projection provenance
→ snapshot managed roots
→ commit receipt-owned files
→ verify + publish receipt

provenance.agent_projection: schema/projector version, canonical set hash, manifest hash i projected tree hash.

Runtime dispatch

workflow asks maister:<role_id>
→ exact ID validation
→ one manifest row
→ receipt + digest verification
→ adapter/version/collision preflight
→ exact host dispatch
→ observed identity check
→ terminal evidence record
→ return result

Support agent Kiro może mieć generated allowlist, ale nigdy nie staje się 29. rolą.

Receipt, provenance and ownership

  • Receipt deklaruje managed_roots z polityką whole_tree lub leaf_set.
  • Inventory identity to root_id + relative path, nie absolutna ścieżka.
  • Kiro leaf set obejmuje wyłącznie manifest/receipt-derived maister-* artefakty.
  • Journal i rollback obejmują wszystkie roots w jednej transakcji; receipt jest publikowany po pełnym verify.
  • Update usuwa tylko stale receipt-owned entries; hand edit daje drift i zero zmian.

Rollout in Dependency Order

  1. Parser/IR, manifest schema, canonical inventory i Advisor equality invariants.
  2. Pure projector, validator, deterministic fixtures i projection provenance.
  3. Materializer integration; usunięcie ręcznych behavior copies i Kiro exemption po closure tests.
  4. managed_roots i Kiro leaf-set z injected-failure rollback tests.
  5. Generated Cursor/Kiro projections; support inventory osobno.
  6. Exact resolver, host adapter ports, execution record i workflow bindings.
  7. Usunięcie legacy Codex TOML ze źródła/init/docs — bez migratora.
  8. Versioned E5/E6 probes, parity baseline update po gates, full release lifecycle.

Concrete Examples

1. Zwykły Advisor na Codex

Given canonical Advisor i receipt-bound manifest, when workflow żąda maister:advisor, then resolver wybiera jeden wiersz, adapter wstrzykuje prompt do generic subagenta i zapisuje standardowy evidence record. Nie odczytuje TOML, readonly ani sandbox branch.

2. Kiro obok agenta użytkownika

Given użytkownik ma personal-reviewer.json, when Maister instaluje 28 par, then receipt obejmuje tylko maister-*, osobisty agent zostaje byte-exact, URI mają closure, a uninstall usuwa tylko owned leaves.

3. Brak exact invocation proof

Given 28/28 materializuje się poprawnie, ale host nie pozwala obserwować exact identity, when zbierane jest evidence, then structural gate może przejść, a E6 pozostaje unavailable. Nie ma native-pass claim ani fallbacku.

Out of Scope

  • Migracja, backup, hash-gated cleanup, preservation i journal dla legacy Codex TOML.
  • Nowy ogólny workflow DSL lub redesign orchestratora.
  • Specjalne uprawnienia, model, readonly, sandbox lub polityka Advisor.
  • Nowe canonical roles poza istniejącymi 28.
  • Jeden wspólny format fizyczny hostów.
  • Uznawanie nieweryfikowalnego native probe za sukces.
  • Przejmowanie całego ~/.kiro lub plików użytkownika.

Success Criteria

  1. Dokładnie 28 ról spełnia identity invariant; Advisor nie ma wyjątku.
  2. Dwa runs projectora dają identyczne paths, bytes, modes, order i tree hash.
  3. Każdy target ma 28/28, support inventory osobno, Kiro ma 100% reference closure.
  4. Full lifecycle zachowuje multi-root ownership i unrelated Kiro files.
  5. Każda logical role daje jeden dispatch plan; błędy kończą się typed error i zero fallback.
  6. Versioned E5/E6 rozróżnia dwie role i Advisor lub jawnie raportuje unavailable.
  7. Repo/init/docs nie zawiera legacy TOML path ani compatibility code; clean packages przechodzą pełny lifecycle.
