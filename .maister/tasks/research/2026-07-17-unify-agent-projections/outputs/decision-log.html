Architecture decision log

Ujednolicona projekcja agentów Maister

Sześć przyjętych decyzji definiujących canonical ownership, projekcję, dispatch, Kiro filesystem ownership, clean installation i evidence.

6ADR total
6accepted
0proposed
0superseded ADRs
1prior recommendation corrected

TL;DR

Przyjęto sześć decyzji: jedno kanoniczne źródło i manifest, projekcję w stagingu, exact fail-closed dispatch, wąskie multi-root ownership Kiro, clean install bez migracji Codex TOML oraz trzy oddzielne poziomy evidence.

Najważniejsza korekta: rezygnacja z hash-gated legacy migration. Stare TOML-e są usuwane ze źródła rozwiązania, a implementacja nie obsługuje upgrade z poprzedniej topologii.

Key Decisions

  • 28 canonical Markdownów jest jedynym właścicielem zachowania.
  • Reprezentacje hostów powstają w stagingu i są digest-bound z receipt/provenance.
  • maister:<role_id> jest rozwiązywany dokładnie, bez fallbacku.
  • Kiro przejmuje tylko jawnie wyliczone native leaf entries.
  • Advisor jest zwykłym wierszem manifestu; legacy TOML nie ma lifecycle.
  • Materialization, discovery i invocation mają osobne evidence.

Open Questions / Risks

  • Codex per-spawn model control, Cursor precedence i obserwowalna Kiro identity pozostają host-version dependent.
  • Multi-root transaction musi zachować byte-exact rollback i race detection.
  • Bez bezpiecznego E5/E6 scenario operational support pozostaje unavailable mimo poprawnej projekcji.
Decisions: ADR-001 · ADR-002 · ADR-003 · ADR-004 · ADR-005 · ADR-006

ADR-001: Jedno kanoniczne źródło ról i wersjonowany manifest

Accepted

Context

Maister posiada 28 spójnych plików Markdown, ale Cursor i Kiro utrzymują osobne behavior-bearing assets, które odbiegły od źródła. Wspólną własnością między hostami jest identity i zachowanie roli, nie format pliku.

Decision Drivers

  • Jeden właściciel portable behavior.
  • Normalized parity zamiast file counts.
  • Jawne, wersjonowane różnice hostów.
  • Canonical digest binding.
  • Brak wyjątku Advisor.

Considered Options

  1. Ręczne zestawy agentów per overlay.
  2. Pełny neutralny DSL generujący też canonical Markdown.
  3. Istniejące canonical Markdown → mały IR + versioned projection manifest.

Decision Outcome

Chosen option: 3. Wykorzystuje istniejące 28 ról jako jedyne źródło, usuwa duplikację i nie dodaje zbędnego DSL. Manifest przechowuje identity, host mapping, transform IDs i tools profiles.

Consequences

Good
  • Jedna zmiana propaguje się do wszystkich targetów.
  • stem=name jest prostym invariantem.
  • Support agents są liczeni osobno.
Bad
  • Parser i manifest są krytycznym build/install path.
  • Każda transformacja potrzebuje wersji, allowlisty i fixture.

ADR-002: Deterministyczna projekcja w stagingu transakcji

Accepted

Context

Generowanie do checkoutu tworzy committed duplicates, a runtime generation omija release closure, receipt ownership, drift detection i rollback. Istniejący materializer ma deterministyczny staging i walidację przed target mutation.

Decision Drivers

  • Immutable source provenance.
  • Reproducible bytes, modes, paths i archives.
  • Wspólny transactional lifecycle.
  • Brak committed prompt copies.
  • Drift detection przed publikacją.

Considered Options

  1. Zatwierdzone generated assets.
  2. Generation per invocation.
  3. Pure projector po source resolution, w stagingu, przed validation/hash/receipt.

Decision Outcome

Chosen option: 3. Włącza projekcję do istniejącej granicy transakcji i release. Provenance zapisuje projector version, canonical set hash, manifest hash i projected tree hash.

Consequences

Good
  • Ten sam candidate jest validated, installed, hashed i packaged.
  • Projection failure nie mutuje targetu.
  • Hand edit daje drift.
Bad
  • Materializer ma nową fazę i typed errors.
  • Receipt/provenance schema wymaga wersji.

ADR-003: Dokładny fail-closed resolver i natywne adaptery

Accepted

Context

Semantic binding labels nie implementują lookupu, dispatch ani result evidence. Natural-language routing lub inline fallback może wykonać niewłaściwą rolę i ukryć brak host support.

Decision Drivers

  • Jednoznaczny logical role selection.
  • Wspólne failure semantics.
  • Natywny mechanizm hosta.
  • Result binding do digestów.
  • Advisor equality.

Considered Options

  1. Natural-language selection hosta.
  2. Inline root-agent execution przy braku subagenta.
  3. Exact resolver + trzy host-native adapters, bez fallbacku.

Decision Outcome

Chosen option: 3. Portable resolver tworzy DispatchPlan. Codex używa generic subagent injection, Cursor exact plugin agent, a Kiro exact custom agent. Każdy adapter zwraca wspólny terminal record.

Consequences

Good
  • Unknown, duplicate, collision, unavailable i mismatch są typed errors.
  • Workflow zna tylko maister:<role_id>.
  • Advisor idzie tą samą ścieżką.
Bad
  • Słabe exact dispatch/identity pozostaje unavailable.
  • Adapter contracts są host-versioned.

ADR-004: Wielokatalogowe, lecz wąskie ownership dla Kiro

Accepted

Context

Kiro odkrywa custom agents pod .kiro/agents lub ~/.kiro/agents; prywatny .kiro-maister nie jest native discovery rootem. Przejęcie całego ~/.kiro naruszyłoby własność użytkownika.

Decision Drivers

  • Native discovery.
  • Preservation of unrelated user files.
  • Atomic lifecycle i byte-exact rollback.
  • Receipt-owned cleanup.
  • Shadow/collision protection.

Considered Options

  1. Tylko .kiro-maister.
  2. Całe ~/.kiro jako Maister-owned.
  3. Jawny kiro_native_agents managed root z receipt-owned leaf set.

Decision Outcome

Chosen option: 3. Receipt używa root_id + relative path, a journal obejmuje primary i native leaf roots w jednej transakcji.

Consequences

Good
  • Native discovery i prompt closure.
  • Uninstall dotyka tylko receipt entries.
  • User agents pozostają poza ownership.
Bad
  • Transaction/recovery/tests obsługują wiele roots.
  • Arbitrary external writer nadal wymaga operator quiescence i rechecks.

ADR-005: Czysta instalacja bez migracji legacy Codex TOML

Accepted corrects prior recommendation

Context

Wcześniejsza rekomendacja proponowała hash-gated migrację .codex/agents/{advisor,arbiter}.toml. Użytkownik doprecyzował, że rozwiązanie będzie instalowane od zera i nie wymaga upgrade ani zachowania starej topologii.

Decision Drivers

  • Clean/from-scratch installation.
  • Minimal implementation i no backward compatibility.
  • Advisor equality.
  • Mniejsza powierzchnia transaction/recovery/docs.

Considered Options

  1. Hash-gated migration z backup/restore.
  2. Best-effort deletion wszystkich legacy TOML.
  3. Usunięcie legacy TOML i generation ze źródła; zero runtime migration/cleanup.

Decision Outcome

Chosen option: 3. Repozytorium, init, release topology i dokumentacja nie zawierają TOML path. Instalator nie ma migratora, preservation branch, cleanup journal ani compatibility flag. Ta decyzja jawnie zastępuje legacy migration recommendation z syntezy/raportu.

Consequences

Good
  • Brak nieużywanego migration code.
  • Mniej schema, tests i failure modes.
  • Codex ma jeden generic injection path.
Bad
  • Upgrade istniejącej legacy installation jest unsupported.
  • Operator non-clean environment przygotowuje go poza tym rozwiązaniem.

ADR-006: Oddzielne dowody materializacji, discovery i invocation

Accepted

Context

Inventory lub descriptor nie dowodzi discovery, a discovery nie dowodzi exact invocation. Niektóre środowiska nie mają executable, authentication, safe adapter lub obserwowalnego scenario.

Decision Drivers

  • Brak fałszywych native support claims.
  • Freshness tied to host/scenario/provenance.
  • Layer-specific diagnostics.
  • Wrong identity fails closed.
  • Zgodność z E1-E6 i unavailable.

Considered Options

  1. Structural validation oznacza full host support.
  2. Discovery + invocation jako jeden probe.
  3. Oddzielne structural/materialization, E5 discovery i E6 exact invocation gates.

Decision Outcome

Chosen option: 3. Brak prerequisite daje unavailable; wrong identity, digest mismatch lub rzeczywisty scenario error daje failed.

Consequences

Good
  • Precyzyjne packaged/discovered/operational claims.
  • Host/source/manifest/scenario changes expire evidence.
  • E6 odróżnia dwie role i Advisor.
Bad
  • Native evidence potrzebuje versioned fixtures/runtime/auth.
  • Target może pozostać provisional mimo structural parity.
