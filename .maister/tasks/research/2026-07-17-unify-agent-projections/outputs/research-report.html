TL;DR

Utrzymaj 28 kanonicznych ról w plugins/maister/agents/*.md, parsuj je do jednego manifestu i projektuj deterministycznie w stagingu transakcji. Codex dostaje generic subagenta z wstrzykniętym promptem, Cursor wygenerowanego plugin-agenta Markdown, a Kiro wygenerowaną parę JSON + prompt w ~/.kiro/agents. Każdy workflow wybiera logiczny role_id; brak, kolizja lub niedostępny runtime kończy się fail-closed. advisor jest zwykłą rolą, bez TOML, readonly, sandboxu i osobnej instalacji.

Key Decisions

  • Jeden behavior owner, jeden manifest i jeden staged projector.
  • Runtime injection dla Codex; natywne projekcje dla Cursor i Kiro.
  • Host helpers są liczone osobno od canonical parity.
  • Legacy Codex TOML znika ze źródła rozwiązania; clean install nie ma ścieżki migracyjnej.

Open Questions / Risks

  • Codex spawn schema/model override, Cursor collision precedence i exact Kiro delegation wymagają native probes.
  • Do ustalenia pozostaje miejsce projection digests w receipt/provenance.

Identity contract

role_id = filename stem = frontmatter.name. Workflow używa maister:<role_id>, a host-facing ID to maister-<role_id>.

Projection contract

Każdy canonical row daje dokładnie jedną projekcję/injection per host. Prompt pochodzi z canonical bytes i named transforms.

Advisor equality

advisor ma ten sam IR, projector, ownership, resolver, adapter i test path. Test odrzuca każdy specjalny TOML, read-only, sandbox lub destination.

Support agents

Cursor/Kiro explore i Kiro maister należą do osobnego support_agents inventory i nie maskują brakującej roli.

Transaction i migration

Managed projections

  • Projector działa przed snapshot/commit.
  • Output wchodzi do receipt i drift detection.
  • Update usuwa tylko stare receipt-owned pliki.
  • Rollback przywraca bytes, modes, symlinks i topologię.

Kiro multi-root

Zachowaj prywatny .kiro-maister, ale journaluj ściśle wyliczony leaf-set maister-*.json i prompts pod ~/.kiro/agents. Nie przejmuj całego ~/.kiro.

Legacy Codex cleanup

Znany historyczny hash: backup i usuń. Modified, unknown, symlink lub unsafe: zero zmian. Failure: exact restore. Migracja jest jawna i idempotentna.

Execution evidence

Zapisuj workflow/work-item, logical role, source digest, external ID, host/version, model status, dispatch ID, attempts i terminal result.

SWOT i confidence

Strength

Kompletny canonical source, deterministyczny materializer i dojrzały receipt/drift/rollback.

Weakness

Brak projectora i executable adapters; Cursor drift; Kiro wrong root i unresolved refs.

Opportunity

Mały IR i staged projector wykorzystują istniejące seams bez redesignu wszystkich workflows.

Threat

Wersjozależne host contracts i ryzyko mylenia structural pass z runtime support.

Confidence: wysoka dla rekomendacji architektonicznej i diagnozy obecnego stanu; operational support pozostaje niezweryfikowany do czasu native probes.

Odrzucone alternatywy

28 native Codex TOML profiles

Nie są plugin component i wprowadzają osobny discovery/collision lifecycle; runtime injection wystarcza.

Ręcznie utrzymywane Cursor/Kiro assets

Zachowują drugiego behavior ownera i drift.

Runtime generation lub modyfikacja checkout

Omija receipt/release/provenance albo łamie immutable source binding.

Natural-language fallback i file-count parity

Nie gwarantują exact identity ani canonical bijection.

Concrete development handoff

  1. Canonical parser, IR, manifest i normalized collision tests.
  2. Pure staged projector, transform registry, digests i --check.
  3. Materializer integration, Kiro reference closure i host projections.
  4. Kiro additional managed leaf root, transaction i receipt/provenance.
  5. Exact resolver, execution record i trzy host adapters.
  6. Usunięcie legacy Codex TOML ze źródła, init i dokumentacji bez migration runtime.
  7. Projection/transaction/native/release tests, potem parity baselines i dokumentacja.

Definition of done: 28/28 na trzech hostach, helpery osobno, zero unresolved refs i undeclared drift, zachowany transactional ownership, exact role mapping, brak specjalnej gałęzi Advisora oraz pinned probes dla dwóch rozróżnialnych ról i Advisora.

Architecture choice: nie pozostają dwie równorzędne alternatywy. Otwarte szczegóły dotyczą schema i evidence, nie kierunku.
